# 工作流编辑器需求与迭代计划

## 目标

把当前前端 Demo 推进成一个可实际试用的类 Dify 工作流编辑器。第一阶段重点不是接入真实模型，而是先让用户能够搭建、保存、恢复、导入、导出并模拟运行工作流。

## 目标用户

- 想用可视化方式编排 AI 应用流程的产品经理、开发者和运营人员
- 想快速验证 AI 工作流结构，但暂时不想先搭后端和数据库的个人开发者

## MVP 范围

当前阶段要支持：

- 可视化节点画布
- 节点新增、拖拽、连线、删除
- 节点配置编辑
- 输出变量展示
- 模拟运行预览
- 本地保存到浏览器
- 刷新页面自动恢复
- JSON 导入和导出
- 重置为示例工作流

暂不支持：

- 用户登录和权限
- 多工作流列表
- 数据库存储
- 真实大模型 API 调用
- 知识库上传和向量检索
- 外部工具真实执行
- 团队协作和版本历史

## 节点类型

- 用户输入：收集用户请求和变量
- 知识检索：模拟从文档或知识库中找上下文
- 大模型：模拟调用模型生成草稿
- 工具调用：模拟调用 API、脚本或内部动作
- 条件分支：模拟根据规则进行路由
- 最终回答：组装并返回最终结果

## 数据结构

工作流定义包含：

- `name`：工作流名称
- `version`：定义版本
- `nodes`：节点 ID、位置和节点配置
- `edges`：连线 ID、起点和终点
- `updatedAt`：本地保存时间

## 当前实现方案

第一阶段仍保持纯前端：

1. 使用 React Flow 管理画布、节点和连线。
2. 使用 React state 管理当前工作流。
3. 使用 `localStorage` 保存最新工作流定义。
4. 使用隐藏文件输入导入 JSON。
5. 使用 Blob 下载导出 JSON。
6. 保存、导入、重置后清理运行预览，避免用户误读旧结果。

## 后续迭代计划

### 第二阶段：真实前端执行器

- 按连线拓扑排序执行节点
- 建立变量上下文
- 支持简单模板变量替换
- 条件节点支持 true/false 分支
- 在运行日志中展示每个节点输入和输出

### 第三阶段：后端 API

- `POST /api/workflows` 保存工作流
- `GET /api/workflows/:id` 读取工作流
- `POST /api/runs` 执行工作流
- 服务端校验工作流结构

### 第四阶段：真实模型与工具

- 大模型节点接入模型 API
- 知识检索节点接入文档或向量库
- 工具节点接入 HTTP 请求或插件执行器
- 增加运行错误处理和重试
